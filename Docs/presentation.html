01 デファードレンダリングパイプライン

IRenderPass インターフェースを定義し、各描画フェーズをプラグイン的に追加できる設計。 描画パスを追加する際にエンジンのコアコードを変更する必要がない。

ShadowRenderPass — Cascaded Shadow Maps
↓
GBufferRenderPass — Albedo / Normal / WorldPos / Material
↓
LightingRenderPass — ライティング計算 (Deferred)
↓
ForwardSceneRenderPass — 前景オブジェクト + 最終合成

ポイント

オブジェクトごとに DrawPath（Deferred / Forward）を選択できる。
半透明オブジェクトや特殊エフェクトはForwardに残し、不透明オブジェクトはGBufferへ。

// IRenderPass.h — 拡張のための抽象インターフェース
class IRenderPass {
public:
    virtual void Execute(SharedRenderTarget& rt) = 0;
    virtual ~IRenderPass() = default;
};

// RenderPipeline.cpp — パスの追加は push_back だけ
passes_.push_back(make_unique<ShadowRenderPass>(ctx));
passes_.push_back(make_unique<GBufferRenderPass>(ctx));
passes_.push_back(make_unique<LightingRenderPass>(ctx));
passes_.push_back(make_unique<ForwardSceneRenderPass>(ctx));

レンダラーのインターフェース分離

IDeferredDrawable と IShadowCaster を別インターフェースに分離。
Object3d がキャッシュリストを保持し、フレームごとの dynamic_cast を排除。

02 敵 AI：State Pattern + Component 設計

ステートが「判断」し、コンポーネントが「実行」する役割分担。 距離ベースの確率テーブルで行動を選択し、同じコンポーネントを複数の敵タイプで再利用できる。 

// アイドル状態での確率的行動選択
if (distance <= 3.0f) {
    // AttackA:10%  SideMove:10%  Move:80%
    RandomSelect({.1f, .1f, .8f}, ...);
} else if (distance <= 16.0f) {
    // AttackB:5%  SideMove:25%  Move:70%
    RandomSelect({.05f, .25f, .7f}, ...);
}

CatmullRom 補間による近接攻撃

EnemyMeleeAttackComponent が武器の制御点列を CatmullRom スプラインで補間。 Windup フェーズ → Attack フェーズと段階を管理し、自然な振り下ろし軌道を実現。

EnemySensorComponent

プレイヤーとの距離・方向をキャッシュ。各ステートが毎フレーム計算する重複を排除。

EnemyMovementComponent

MoveToward / MoveAway / MoveSideways / Stop を提供。velocity を直接書き換えるシンプルな設計で、複数の敵タイプが共有。

EnemyMeleeAttackComponent

IsFinished() / IsWindingUp() でステート遷移条件を問い合わせ。

実装例：Hellkaina / GruntMelee

03 GPU リソース管理（ハンドル方式 + Staged Release）

DirectX 12 では CPU が Release() を呼んでもGPUがまだそのリソースを参照中の場合、クラッシュが発生する。 ハンドルによる間接参照 と フェンス同期による遅延解放 でこれを防ぐ。 

// ハンドルは 32bit 整数 — 実体と切り離す
using BufferHandle = uint32_t;

// 解放要求 → ステージングキューへ積むだけ
void ReleaseBuffer(BufferHandle handle) {
    staging_.push(handle); // 即座には解放しない
}

// フレーム終了時にフェンス値と紐付け
void OnFrameEnd(uint64_t fenceValue) {
    pending_.push({fenceValue, staging_.pop()});
}

// GPU 完了後に実際の D3D Release
void ProcessPendingReleases(uint64_t completedFence) {
    while (pending_.front().fence <= completedFence)
        pending_.pop(); // ここで ComPtr が解放される
}

解放フロー

ReleaseBuffer(handle) 呼び出し
↓ フレーム内バッファ
staging_ キュー — まだ実体は生存
↓ OnFrameEnd()
pending_ リスト — (fenceValue, resource)
↓ GPU フェンス通過後
ProcessPendingReleases() → 実際の解放

デバッグ時のメモリリーク検出

D3DResourceLeakChecker が DXGI Debug Layer を利用し、デストラクタ時に未解放 D3D リソースを自動でログ出力。リリースビルドでは完全に除去される。

04 武器トレイルエフェクト（動的リボンメッシュ）

刀身の先端（Tip）と手元（Hilt）の軌跡を毎フレーム記録し、双方を結んだリボン状のメッシュをCPU側で動的生成してGPUへ転送。 専用の Trail PSO で半透明合成する。 

// 毎フレーム: 先端・根元・経過時間を deque に追加
struct TrailPoint {
    Vector3 tip, hilt;
    float   age;        // フェードアウト用
};
deque<TrailPoint> points_; // 最大 32 点

// 隣接 2 点 → 4 頂点の四角形に展開してリボンを形成
for (int i = 0; i < points_.size()-1; ++i) {
    vertices[i*4+0] = points_[i  ].tip;
    vertices[i*4+1] = points_[i  ].hilt;
    vertices[i*4+2] = points_[i+1].tip;
    vertices[i*4+3] = points_[i+1].hilt;
}

なぜ動的 VB か

トレイルの形状はフレームごとに変化するため、Upload ヒープに置いた頂点バッファを CPU から直接書き換え（Map/Unmap）。固定サイズ VB の再利用でアロケーションゼロ。

頂点レイアウト

Position: R32G32B32_FLOAT
TexCoord: R32G32_FLOAT （U=位置、V=Tip/Hilt 判別）
Color: R32G32B32A32_FLOAT （age で alpha フェード）

ライフタイム管理

各点は age フィールドを持ち、経過時間で opacity を線形補間。 古い点は deque の先頭から自動削除。攻撃終了後もフェードアウトが自然に完了する。

パーティクルシステムとの分離

トレイルは独立した Trail PSO を使用し、通常のパーティクルと描画パスが干渉しない。 ブレンドモードを別個に設定できるため視覚表現の自由度が高い。

05 コンボシステム（攻撃グラフ）

攻撃の連鎖を 有向非巡回グラフ（DAG） で表現。 各ノードは「次に繋げられる攻撃リスト」を持ち、入力条件（ボタン・スティック方向・ロックオン状態）に応じて遷移先を動的に決定する。 

// 攻撃ノードの定義
struct AttackNode {
    string              name;
    vector<AttackEdge>  nextAttacks; // DAG の辺
};
struct AttackEdge {
    InputCondition condition; // ボタン + 方向 + ロックオン
    AttackNode*    next;
};

// 攻撃データ：モーション制御点 + タイミング + 効果
struct AttackData {
    vector<Vector3> controlPoints; // CatmullRom 用
    float preDelay, attackDuration;
    float postDelay, nextAttackDelay;
    float damage, knockbackPower;
    HitStopConfig hitStop; // 時間停止パラメータ
    ReactionType  reaction; // HitStun / Knockback / Launch
};

CatmullRom スプライン補間

攻撃モーションの軌道を制御点の配列で定義し、CatmullRom 曲線で補間。 直線的でなく自然な弧を描く攻撃動作を、少ないデータ量で表現できる。

HitStop（時間停止エフェクト）

HitStop クラスが時間スケール係数（0.0 = 完全停止）を管理。 命中時に数フレームだけゲーム時間を遅らせ、打撃感を演出。 フレームスキップでなくスケール乗算なので物理計算との整合性を保つ。

JSON 外部定義

攻撃データは GlobalVariables 経由で JSON から動的ロード。 コードを再コンパイルせずにコンボ構成やダメージ値の調整が可能。

06 コリジョン状態追跡（Enter / Stay / Exit）

単純な「接触しているか否か」ではなく、前フレームとの差分を取ることで OnCollisionEnter / OnCollisionStay / OnCollisionExit の三種類のイベントを正確に発火させる。 

// 毎フレーム: 現在の衝突ペアを収集
set<CollisionPair> currentCollisions_;
set<CollisionPair> previousCollisions_;

for (auto& pair : currentCollisions_) {
    if (!previousCollisions_.count(pair))
        pair.NotifyEnter();   // 新規 → Enter
    else
        pair.NotifyStay();    // 継続 → Stay
}
for (auto& pair : previousCollisions_) {
    if (!currentCollisions_.count(pair))
        pair.NotifyExit();    // 消滅 → Exit
}
previousCollisions_ = currentCollisions_;

対応コリジョン形状

  • AABB — 軸並行バウンディングボックス
  • Sphere — 球体（キャラクター・武器攻撃判定）
  • OBB — 方向付きバウンディングボックス（地形・障害物）

カテゴリフィルタリング

各コライダーは CollisionCategory（Player / Enemy / PlayerWeapon / EnemyWeapon / Ground）を持つ。 判定前にカテゴリ対の組み合わせを確認し、不要なペアを早期除外してブロードフェーズのコストを削減。

ゲームロジックへの活用

武器が敵に当たった瞬間（Enter）だけダメージ処理。Stay 中は二重ヒットしない。 Exit でヒットフラグをリセット → 素直で明確なゲームロジックが書ける。

07 PSO 管理のリファクタリング

PSOManager.cpp が14種類のパイプライン定義をすべて内包し、2138行の神クラスになっていた。 単一責任原則を適用し、各 PSO タイプを専用の PSOBuilder クラスに分離。

Before
2138 lines / PSOManager.cpp
  • 全 PSO を1ファイルに定義
  • 新 PSO 追加 → 巨大ファイル編集
  • PSO ごとのテストが困難
After
270 lines / PSOManager.cpp
  • 14 個の PSOBuilder クラスに分散
  • 新 PSO → 新ファイルを追加するだけ
  • 各 Builder を個別にテスト可能

同時期のシングルトン近代化

18 クラスの static T* ポインタを unique_ptr<T> に置き換え（Meyers' Singleton）。 コピーセマンティクスを明示的に削除し、所有権モデルを明確化。

EngineContext による依存性注入

全サービスを集約した EngineContext 構造体を導入。 RenderPath サブシステムは GetInstance() 呼び出しをゼロにし、テスト容易性と結合度低下を実現。

DirectXManager の責務分離

ShaderCompiler と FrameTimer を抽出し、320 行のデッドコードを削除。 各クラスが単一の関心事を持つ設計へ。
